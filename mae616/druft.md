

# 概要
- 作成アプリ: X風 相互フォロー＋文章投稿プラットフォーム (SNS)
- 開発時間: 40分
- 設計書: 不要
- 環境: ローカル環境のみ
- 品質: プロトタイプ品質 (デザイン、機能ともにMVP)
- UIの言語: 日本語
- デザイン: Pencilファイルを新規作成して画面設計を行う
- アプリのタイトルは画像でなく文字で表記する
- 重要！！！アプリのメインブランチ名は 「mae616」です。絶対に最後出来上がるのは「mae616」でそこから派生したブランチを作って作業してください。それ以外のブランチに許可なくマージしないでください


# 技術スタック
## フロントエンド
-	React（latest）+ Vite + TypeScript + Tailwind CSS
-	状態管理: useState / useEffect
-	ルーティング: react-router-dom
-	認証: なし（viewerId固定。例: const VIEWER_ID = 1）

静的解析 / 整形
- ESLint（TypeScript対応、recommended設定）
- Prettier（コード整形専用）

注意点
-	グローバル状態管理は入れない（Redux等は使わない）
-	viewerIdはログインの代替。UIにユーザー切替を入れるならstate管理にする
-	自分自身（viewerId）にはフォローボタンを表示しない
-	APIレスポンスをそのままstateに入れ、派生計算は極力バックエンド側に寄せる

## バックエンド（REST API）
-	Express.js（latest）
-	cors 有効化
-	bodyParser(json)

注意点
-	フォローはトグルAPIにする（存在確認→削除 or 追加）
-	タイムラインは「viewerがフォロー中のユーザー」の投稿のみ返す
-	createdAt降順で返却
-	countsはAPIで計算して返却
-	レスポンスは必要最小限

## DB
-	SQLite
-	ライブラリ: better-sqlite3

注意点
-	起動時にテーブルがなければ作成
-	seedデータを必ず5人以上用意
-	followsは複合ユニーク制約（followerId + followeeId）
-	createdAtはISO文字列で保存
-	外部キー制約は時間がなければ省略可

## seedデータ
- users: 5〜6人
- posts: 各ユーザーに1〜2件あるとタイムライン確認が楽
- follows: 何人か相互フォロー状態を事前に作っておく

注意点
- viewerId=1が最低1人はフォロー済みにしておくとデモが安定
- 空タイムライン状態も起こり得るのでseedで調整

## テスト方針
-	Playwright等のE2Eテストは実施しない
-	フロントエンドの結合テストは実施しない
-	包括的な単体テストは実施しない
-	最重要ケース1本のみバックエンドで検証する

実施する最重要テスト（バックエンド）
-	/timeline APIが「viewerがフォローしているユーザー」の投稿のみをcreatedAt降順で返すことを確認する

## 仕様（完成条件）
-	ユーザー一覧表示 → 5人以上表示される
-	フォロー/アンフォロー. → トグル動作し、UIとDBが一致
-	文章投稿 → DB保存され即時表示
-	タイムライン → フォロー中のみ、createdAt降順
-	プロフィール画面 → ユーザー情報＋その投稿一覧
-	フォロー数/フォロワー数 → DBから正しく集計
-	タイムライン/ユーザー一覧 → ユーザー名からユーザー画面に遷移できる
