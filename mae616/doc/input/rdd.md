# 🧑‍💻 ユースケース
## コンテキスト（AI用）
- シナリオ1: ユーザー一覧を見て、気になるユーザーをフォロー/アンフォローする
- シナリオ2: テキストを投稿し、タイムラインに表示される
- シナリオ3: タイムラインでフォロー中ユーザーの投稿を時系列で閲覧する
- シナリオ4: ユーザー名をクリックしてプロフィール画面に遷移し、そのユーザーの投稿一覧を見る
- MVP範囲: 全シナリオがMVP

## 人間レビュー用
- **ユーザー一覧** → フォローボタン押下 → フォロー状態トグル
- **投稿フォーム** → テキスト入力 → 送信 → 即時表示
- **タイムライン** → フォロー中ユーザーの投稿のみ表示（新しい順）
- **プロフィール** → ユーザー情報 + 投稿一覧 + フォロー数/フォロワー数

# 🏗️ 機能要件
## コンテキスト（AI用）
- 機能一覧（優先度順）:
  1. **ユーザー一覧表示** — 5人以上表示される
  2. **フォロー/アンフォロー** — トグル動作し、UIとDBが一致。自分自身にはフォローボタンを表示しない
  3. **文章投稿** — DB保存され即時表示
  4. **タイムライン** — フォロー中ユーザーの投稿のみ、createdAt降順
  5. **プロフィール画面** — ユーザー情報＋その投稿一覧
  6. **フォロー数/フォロワー数** — DBから正しく集計
  7. **ユーザー名リンク** — タイムライン/ユーザー一覧からユーザー画面に遷移できる
- DoD（完了条件）: 上記7項目すべてが動作すること

## 人間レビュー用
- アプリのタイトルは**画像ではなく文字で表記**する
- UIの言語は**日本語**
- 認証なし（viewerId固定: `const VIEWER_ID = 1`）
- viewerIdはログインの代替。自分自身（viewerId）にはフォローボタンを表示しない

# ⚙️ 技術スタック
## コンテキスト（AI用）

### フロントエンド
- 言語/ランタイム: TypeScript + React (latest) + Vite
- スタイリング: Tailwind CSS
- 状態管理: useState / useEffect（グローバル状態管理は入れない。Redux等は使わない）
- ルーティング: react-router-dom
- 認証: なし（`const VIEWER_ID = 1` で固定）
- 静的解析: ESLint（TypeScript対応、recommended設定）
- 整形: Prettier

### バックエンド（REST API）
- フレームワーク: Express.js (latest)
- ミドルウェア: cors有効化, bodyParser(json)
- フォロー: トグルAPI（存在確認→削除 or 追加）
- タイムライン: viewerがフォロー中のユーザーの投稿のみ返す。createdAt降順
- counts: APIで計算して返却
- レスポンス: 必要最小限

### DB
- SQLite + better-sqlite3
- 起動時にテーブルがなければ作成
- followsは複合ユニーク制約（followerId + followeeId）
- createdAtはISO文字列で保存
- 外部キー制約は時間がなければ省略可

### seedデータ
- users: 5〜6人
- posts: 各ユーザーに1〜2件
- follows: 何人か相互フォロー状態を事前に作る
- viewerId=1が最低1人はフォロー済み

### テスト戦略
- E2Eテスト: 実施しない
- フロントエンド結合テスト: 実施しない
- 包括的な単体テスト: 実施しない
- **実施する最重要テスト（バックエンド）**: `/timeline` APIが「viewerがフォローしているユーザー」の投稿のみをcreatedAt降順で返すことを確認する

## 人間レビュー用
- APIレスポンスをそのままstateに入れ、派生計算は極力バックエンド側に寄せる
- viewerIdはログインの代替。UIにユーザー切替を入れるならstate管理にする
