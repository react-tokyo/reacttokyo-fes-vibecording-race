# チーム連携プロトコル

AIエージェントチームの運用規約を定義する。
Git運用の詳細は `doc/guide/git_workflow.md` を参照（重複定義しない）。

## メンテナンス情報
- **最終更新日**: 2026-02-25
- **対象**: ai-template を使用するすべてのプロジェクト
- **前提**: `doc/guide/git_workflow.md` のブランチ構造・CI設定に準拠

---

## 1. チームの目的

AIチームは「実装部隊」ではなく **ユーザーちゃんの思考を広げる多面鏡＋構造化装置** として機能する。

- 各メンバーは「同意」より **別角度からの指摘・懸念・代替案** を優先して出す
- 出力は散漫な報告ではなく **構造化されたフォーマット** で返す
- 最終判断は常にユーザーちゃん（Human-in-the-loop）

---

## 2. 役割定義

### 2.1 ユーザーちゃん（最終決裁者）
- すべての Decision Gate で最終判断
- Sprint完了時の動作確認
- レビュー判断の最終承認

### 2.2 リーダーにゃんこ 👑（Opus / 進行役）
- **モデル**: Opus（フルスタック統合視点）
- **責務**: タスク分割・担当割り当て・進捗監視・統合判断
- **権限**: Sprint PR のレビュー＋マージ判定（ユーザーちゃん動作確認後）
- **制約**: 勝手に決めない。Decision Gate に該当する判断は必ずユーザーちゃんに確認
- **副次**: レビュー指摘の一次判断（Critical/Warning の分類と対応方針提案）

### 2.3 フロントエンドにゃんこ 🎨（Sonnet）
- **専門**: フロントエンド（UI/UX/状態管理/描画）
- **視点の強み**: 「ユーザーから見て変じゃない？」
- **フルスタック理解**: あり（BE/Infraのコードも読める。クロスレビュー可能）

### 2.4 バックエンドにゃんこ ⚙️（Sonnet）
- **専門**: バックエンド（API/DB/ビジネスロジック）
- **視点の強み**: 「仕様ロジックに穴ない？」
- **フルスタック理解**: あり（FE/Infraのコードも読める。クロスレビュー可能）

### 2.5 インフラにゃんこ 🏗️（Sonnet）
- **専門**: インフラ/テスト/CI/CD/デプロイ
- **視点の強み**: 「壊すとしたらどこ？運用で困らない？」
- **フルスタック理解**: あり（FE/BEのコードも読める。クロスレビュー可能）

## 3. GitHubフロー連携

> ブランチ構造・CI設定・ライフサイクルの詳細 → `doc/guide/git_workflow.md`

### 3.1 タスク単位（task/* → sprint/*）

| 項目 | ルール |
|------|--------|
| **PR作成** | 実装完了 + Quality Gate 通過後、担当メンバーが作成 |
| **base** | `sprint/*`（※ `main` ではない） |
| **レビュー** | Sonnet クロスレビュー（隣のメンバーが担当） |
| **マージ** | レビュー指摘の対応完了後、レビュー担当メンバーがマージ |
| **Issue close** | PR本文に `Closes #N` を必ず含める（マージ時に自動close） |

### 3.2 Sprint単位（sprint/* → main）

| 項目 | ルール |
|------|--------|
| **PR作成** | 全タスクマージ後、リーダーが作成 |
| **base** | `main` |
| **レビュー** | Opus deep-review（統合観点：設計整合・リスク・回帰） |
| **動作確認** | **ユーザーちゃんが実施**（Opusレビュー完了後） |
| **マージ** | ユーザーちゃんの動作確認OK後、リーダーがマージ |
| **Milestone close** | マージ後、リーダーが `gh api` で Milestone を close |

### 3.3 Milestone 自動close

```bash
# Sprint PRマージ後にリーダーが実行
MILESTONE_NUMBER=$(gh api repos/{owner}/{repo}/milestones --jq '.[] | select(.title=="sprint-N") | .number')
gh api -X PATCH repos/{owner}/{repo}/milestones/${MILESTONE_NUMBER} -f state=closed
```

## 4. ワークフロー定義

各フローはチーム時と非チーム時で担当が変わる。
非チーム時（single-agent）は従来通り1つのagentが全ステップを実行する。

### 4.1 タスクフロー

#### チーム時

```
リーダー: /task-list
  │  Sprint計画 + Milestone作成
  │  統合ポイントを1つ以上設定
  │  タスクを「動かせる単位」（FE+BE+Infraが揃う縦スライス）で分割
  │  担当メンバーを割り当て
  ↓
各メンバー: /task-detail
  │  自分の担当タスクを詳細化
  │  依存関係（blockedBy/blocks）を設定
  │  team-board.md に状態を記録
  ↓
各メンバー: /task-run
  │  task/* ブランチで実装
  │  Quality Gate 通過（lint + typecheck）
  │  PR作成（base: sprint/*）
  │  PR本文に Closes #N を含める
  ↓
Sonnet クロスレビュー: /basic-review
  │  FE→BE、BE→Infra、Infra→FE のローテーション
  │  指摘あり → 修正 → /basic-review + /deep-review 再実行
  │  指摘なし → レビュー担当がマージ（sprint/*へ）
  │  Issue 自動close
  ↓
全タスク完了後
  ↓
リーダー(Opus): /deep-review
  │  Sprint統合レビュー（設計整合・リスク・回帰）
  │  Sprint PR作成（base: main）
  ↓
ユーザーちゃん: 動作確認
  │  OK → リーダーがマージ（mainへ）
  │  NG → フィードバック → feature_fix/* で修正 → 再レビュー
  ↓
リーダー: Milestone close
  │  gh api で Milestone を closed に更新
```

#### 非チーム時

従来通り: `/task-list` → `/task-detail` → `/task-run` → `/basic-review`
※ PRのbase先は Sprint運用中なら `sprint/*`、なければ `main`

### 4.2 バグフロー

#### チーム時

```
バグ発見（Sprint統合テスト / ユーザーちゃんの動作確認 / CI失敗）
  ↓
リーダー: /bug-new
  │  トラブルシューティングログ生成
  │  担当メンバーを割り当て（影響レイヤーに近い人）
  │  team-board.md に記録
  ↓
担当メンバー: /bug-investigate
  │  調査実行 + 仮説の絞り込み
  │  team-board.md に調査結果を記録
  ↓
担当メンバー: /bug-propose
  │  修正策を列挙（提案＋根拠＋リスク＋別案）
  │  → リーダーに報告
  │  → Decision Gate 該当なら → ユーザーちゃんに確認
  ↓
担当メンバー: /bug-fix
  │  feature_fix/* ブランチで修正実装
  │  Quality Gate 通過
  │  PR作成（base: sprint/*）
  ↓
Sonnet クロスレビュー: /basic-review
  │  指摘対応 → 再レビュー
  │  完了 → マージ（sprint/*へ）
```

#### 非チーム時

従来通り: `/bug-new` → `/bug-investigate` → `/bug-propose` → `/bug-fix`

### 4.3 レビューフロー

#### チーム時（タスクレビュー）

```
タスクPR作成
  ↓
Sonnet クロスレビュー: /basic-review
  │  観点: 仕様逸脱、境界条件、命名、テスト漏れ
  ↓
指摘あり？
  ├── なし → マージ
  └── あり → 担当メンバーが修正
              ↓
            /basic-review + /deep-review 再実行
              ↓
            指摘の判断（§9.3 参照）
              ├── Critical → 必ず修正 → 再レビュー
              ├── Warning → リーダー一次判断
              │     ├── 対応可能 → 修正 → 再レビュー
              │     └── 対応困難 → Issue化（ユーザーちゃん最終確認）
              └── Suggestion → Issue化 or メモ
              ↓
            全指摘対応完了 → マージ
```

#### チーム時（Sprintレビュー）

```
全タスクが sprint/* にマージ済み
  ↓
リーダー(Opus): /deep-review
  │  観点: 設計整合、レイヤー間の整合、リスク、回帰、ロールバック可否
  │  Sprint PR作成（base: main）
  ↓
指摘あり？
  ├── Critical → feature_fix/* で修正 → 再レビュー
  ├── Warning → リーダー判断 → ユーザーちゃん最終確認
  └── なし → ユーザーちゃんの動作確認へ
  ↓
ユーザーちゃん: 動作確認
  ├── OK → リーダーがマージ
  └── NG → フィードバック → 修正サイクル
```

### 4.4 PR対応フロー

#### チーム時

```
PRにレビューコメント受信
  ↓
リーダー: コメントを確認、担当メンバーを割り当て
  ↓
担当メンバー: /pr-respond
  │  1件ずつ対応＋コミット（中断時対策）
  │  全件対応完了
  ↓
/basic-review 再実行
  ↓
指摘なし → マージ
```

#### 非チーム時

従来通り: `/pr-respond` → 1件ずつ対応＆コミット → push

## 5. Sprint設計（ハイブリッド型）

### 5.1 統合ポイント

各Sprintに **統合ポイント** を1つ以上必ず設ける。

- **統合ポイント** = 全レイヤー（FE + BE + Infra）がつながって動作確認できる機能
- ユーザーちゃんの動作確認は統合ポイント単位で実施

### 5.2 先行タスク

統合ポイント以外のタスクは先行実装OK。

- 次Sprintの先行作業として実装可能
- ただし Sprint PR には含めない（次Sprint のブランチに移す or cherry-pick）

### 5.3 フルスタック並行時の区切り

```
Sprint-1: 「ログイン機能」（統合ポイント）
  ├── FE: ログインUI       ← 統合ポイント必須
  ├── BE: 認証API          ← 統合ポイント必須
  ├── Infra: デプロイ設定   ← 統合ポイント必須
  ├── FE: ダッシュボード骨格 ← 先行タスク（次Sprint向け）
  └── BE: ユーザーAPI       ← 先行タスク（次Sprint向け）
```

**一段落の判定基準**: 統合ポイントに関わる全タスクが sprint/* にマージ済み

## 6. Decision Gate

以下の条件に該当する場合、**必ずユーザーちゃんに確認**してから実行する。

| 条件 | 理由 |
|------|------|
| 依存パッケージの追加・変更 | サプライチェーンリスク |
| アーキテクチャ変更 | 影響範囲が広い |
| DB スキーマ変更 | データ破壊リスク |
| セキュリティに関わる変更 | 脆弱性リスク |
| 大規模リファクタ（3ファイル以上） | 回帰リスク |
| RDD（要件定義）からの逸脱 | 仕様ズレ |
| 想定外のブロッカー | 方針判断が必要 |

**確認方法**: tmux のサブペインから通知（メイン作業を止めない）

## 7. Output Contract

各メンバーの出力は **「結論」ではなく以下の構造** で返す。

```markdown
## 提案
{何をしたいか / 何を見つけたか}

## 根拠
{なぜそう判断したか / 参照した情報}

## リスク・懸念
{この判断で起きうる問題}

## 別案
{他の選択肢とトレードオフ}

## 問い
{ユーザーちゃんに確認したいこと / 次に判断が必要なこと}
```

**目的**: ユーザーちゃんの「自分にない観点が出る」「構造が見える」を実現する。

## 8. Handoff（チームホワイトボード）

### 8.1 共有ファイル

`.claude/team-board.md` をチーム全体のホワイトボードとして使う。

**用途**:
- メンバー間の意図・判断の背景の共有
- ブロッカーの可視化
- 依存関係の状態更新
- 非同期の引き渡し情報

### 8.2 フォーマット

```markdown
# Team Board

## 更新日時: YYYY-MM-DD HH:MM

### 現在のSprint
- Sprint名: {sprint-N}
- 統合ポイント: {機能名}
- 状態: {planning / in_progress / review / done}

### メンバー状況
| メンバー | タスク | 状態 | ブロッカー |
|---------|--------|------|-----------|
| 🎨 FE  | #N {概要} | {doing/done/blocked} | {なし / 内容} |
| ⚙️ BE  | #N {概要} | {doing/done/blocked} | {なし / 内容} |
| 🏗️ Infra | #N {概要} | {doing/done/blocked} | {なし / 内容} |

### 引き渡しメモ
- [{from} → {to}] {内容}（{日時}）

### Decision Gate（確認待ち）
- [ ] {確認内容}（{誰が待ち}）
```

### 8.3 更新ルール

- **状態変更時に必ず更新**（タスク開始、完了、ブロッカー発生）
- **引き渡し時に必ずメモを残す**（意図・判断の背景・未確定事項）
- リーダーは **定期的に全体を確認** して整合性を保つ

## 9. Review Rule

### 9.1 レビュー2段構成

| 段階 | 実施者 | 目的 | 観点 |
|------|--------|------|------|
| **タスクレビュー** | Sonnet クロス | 事故落とし | 仕様逸脱、境界条件、命名、テスト漏れ |
| **Sprintレビュー** | Opus | 統合判断 | 設計整合、リスク、回帰、ロールバック可否 |

### 9.2 クロスレビューのローテーション

```
FE → BE のコードをレビュー（仕様ロジック観点）
BE → Infra のコードをレビュー（運用観点）
Infra → FE のコードをレビュー（壊れにくさ観点）
```

全員フルスタック理解ありのため、担当外のコードもレビュー可能。

### 9.3 レビュー指摘の判断フロー

```
指摘発生
  ├── Critical（バグ/セキュリティ/仕様逸脱）
  │     → 必ず修正
  │     → 修正後 basic-review + deep-review を再実行
  │
  ├── Warning（設計改善/可読性/テスト不足）
  │     → リーダーが一次判断
  │     │   ├── 対応可能 → 修正（再レビュー）
  │     │   └── 対応困難 → Issue化して次Sprint
  │     → ユーザーちゃんが最終確認
  │
  └── Suggestion（好み/最適化/将来改善）
        → Issue化 or メモ（今は対応しない）
```

### 9.4 再レビュー条件

修正が **1件でもあれば** basic-review + deep-review を再実行する。

## 10. Quality Gate（自動ガードレール）

> CI設定の詳細 → `doc/guide/git_workflow.md` の「CI設定の段階分け」

| ゲート | タイミング | 通過条件 |
|--------|-----------|---------|
| **コミット前** | 各メンバーの実装完了時 | lint + typecheck 通過 |
| **PR作成前** | タスクPR作成時 | lint + typecheck 通過 |
| **Sprint PR** | Sprint → main PR時 | lint + typecheck + build + test 全通過 |

**ルール**:
- Quality Gate を通過しないコードは PR にしない
- CI結果は PR本文に添付（レビュー効率化）
- 手動での Gate スキップは禁止（`--no-verify` 等）

## 11. Blocker Protocol

```
メンバーが詰まる
  ↓
team-board.md にブロッカーを記録（状態: blocked）
  ↓
リーダーに通知
  ↓
リーダーが判断
  ├── 技術的問題 → 他メンバーに支援依頼 or 自分で解決
  ├── 仕様/方針不明 → ユーザーちゃんに確認（tmuxサブペイン経由）
  └── 外部依存 → 一旦スキップ、別タスクに切り替え
  ↓
解決後 team-board.md を更新（状態: doing）
```

**タイムアウト**: 一定時間（目安: 10分）解決しない場合、リーダーに自動エスカレーション

## 12. 並行作業ルール

### 12.1 非チーム時（single-agent）

| 項目 | ルール |
|------|--------|
| **worktree** | 独立タスクは git worktree で並行実装OK |
| **バックグラウンド** | テスト/ビルドはバックグラウンド実行OK |
| **確認待ち** | tmuxサブペインで確認依頼。メインはブロックしない |

### 12.2 チーム起動時

| 項目 | ルール |
|------|--------|
| **リソース優先** | チームタスク > 並行タスク |
| **並行処理の制御** | チーム起動でリソースが不足する場合、並行処理を一時停止 |
| **復帰** | チーム終了後、停止していた並行処理を再開可能 |

### 12.3 tmux活用

```
┌─────────────────────┬──────────────────┐
│                     │ 確認待ち/通知     │
│  メイン作業ペイン    │ Decision Gate    │
│  （実装が進む）      │ ブロッカー通知    │
│                     │ → ユーザーちゃんY/N   │
├─────────────────────┼──────────────────┤
│  並行タスクペイン    │ 進捗/ログ        │
│  （バックグラウンド） │                  │
└─────────────────────┴──────────────────┘
```

**原則**: 確認待ちでメインの作業を止めない。サブペインで非同期に処理する。

### 12.4 並行処理の権限確認ブロック防止

並行セッション（worktree/バックグラウンド）が権限確認で止まり、ユーザーちゃんに気づかれないまま放置される問題を防ぐ。

**起動時のルール**:
- 並行タスクを起動する前に、**メインセッションでユーザーちゃんに通知**する
  - 「サブペインで {タスク名} を並行実行します。権限確認が出たら {ペイン名} を確認してください」
- 可能な場合は `--dangerouslySkipPermissions` や `acceptEdits` モードで起動し、権限確認を最小化する
  - ただし **セキュリティに関わる操作（依存追加・外部API呼び出し・ファイル削除等）は必ず確認**

**実行中のルール**:
- 並行セッションが権限確認で止まった場合、**メインセッション側がそのことをユーザーちゃんに伝える**
  - 「サブペイン {ペイン名} で確認待ちが発生しています」
- 自分で解決できる場合（安全な操作）はリーダーが代理承認
- Decision Gate に該当する場合はユーザーちゃんに確認

**止まったときのフロー**:
```
並行セッションが権限確認で停止
  ↓
メインセッションがユーザーちゃんに通知
  ↓
ユーザーちゃんが判断
  ├── 承認 → サブペインで承認操作
  ├── 拒否 → 代替案を検討
  └── 不要と判断 → 並行セッションをキャンセルし、メインで再実行
```

## 13. エージェント間の連携ルール

### 13.1 コンテキスト共有

| 情報の種類 | 共有方法 |
|-----------|---------|
| コードの状態 | ファイルシステムで直接確認（git pull / ファイル読み） |
| 意図・判断の背景 | `.claude/team-board.md`（ホワイトボード） |
| ブロッカー | team-board.md + リーダー経由 |
| タスク依存関係 | team-board.md + GitHub Issue の blockedBy/blocks |

### 13.2 通信ルール

- **メンバー → リーダー**: 完了報告、ブロッカー通知、Decision Gate 該当の確認依頼
- **リーダー → メンバー**: タスク割り当て、方針伝達、支援依頼
- **メンバー ↔ メンバー**: 直接通信なし。リーダー経由 or team-board.md
- **リーダー → ユーザーちゃん**: Decision Gate の確認、Sprint レビュー報告

### 13.3 自律と制約のバランス

| 自律でOK | 必ず確認 |
|---------|---------|
| 担当ファイル内の実装判断 | Decision Gate に該当する変更 |
| テスト追加・修正 | 担当外ファイルへの変更 |
| リファクタ（担当範囲内） | 新規依存の追加 |
| バグ修正（担当範囲内） | 仕様解釈に迷った場合 |
