# 開発フローガイド

開発ライフサイクル全体を1本で通すガイドライン。AIが「暗黙の開発フロー」を推測して局所最適で動くことを防ぎ、各フェーズの目的・成果物・スキルの関係を明示する。

## フロー概要

```
要件定義 → デザイン → 開発 → ユーザー確認 → 指摘修正 → ドキュメント整備 → デプロイ
                                                ↑
                                          バグ修正（任意タイミングで発動）
```

---

## 1. 要件定義

- `/project-init` → `doc/input/rdd.md` 作成
- 技術スタック決定（UI言語含む）
- ガードレール決定（リンター/テストライブラリ/Playwright等）
- **RDDが以降の全フェーズの正**（デプロイ手順・技術スタック等もRDDに記載があればそちら優先）
- **プロジェクト作成は公式推奨方法を使う**: 技術スタックの公式が推奨するプロジェクト作成方法（`npm create`、`npx create-*-app`等のボイラーテンプレート）がある場合はそれを使う

---

## 2. デザイン

デザインワークフローの詳細は `doc/guide/design_workflow.md` を参照（Pencil / Figma / mock、AIが起点を自動判断）。

### デザインソース参照の優先順位

タスク実行（`/task-run`）やフロントエンド/バックエンド実装で、UIに関わる作業をする場合は**デザインソースを参照してから実装に入る**。デザインを参照せずに作ると、画面構成やインタラクションの意図からズレた実装になりやすい。

| 優先度 | ソース | 参照方法 |
|--------|--------|---------|
| 1 | **Pencilデザインファイル**（`doc/input/design/*.pen`） | Pencil MCP で画面構成・コンポーネント・インタラクションを参照 |
| 2 | **HTMLモック**（`doc/input/design/html/`） | ブラウザまたはコードで画面構成を参照 |
| 3 | **Figma** | Figma MCP（Dev Mode）で参照 |
| 4 | **なし** | `doc/input/rdd.md` の機能要件から操作フローを推定し、短問で確認 |

> Pencilデザインファイルは要件定義（`rdd.md`）の後に作成される。ある場合は必ず参照すること。

### 対象スキル（この参照ルールが適用される場面）
- `/task-detail`（Issue詳細化）: 実装ブループリント作成時にデザインを参照
- `/task-run`（Issue実行）: 実装着手時にデザインを参照
- `frontend-implementation`（判断軸）: デザイン→実装の翻訳時
- `developer-specialist`（判断軸）: UI関連の実装時

### ブランチ
- デザインフェーズ完了後、開発着手前にブランチを切る

## 3. 開発（チーム or シングル）

### タスク管理（省略不可）

`/task-list` → `/task-detail` → `/task-run` を**必ずこの順で実行**する。

| スキル | シングル | チーム |
|--------|---------|--------|
| `/task-list` | 本人が実行 | チームリーダーが実行 |
| `/task-detail` | Issue詳細化（デザインソースも参照して追記） | 同左 |
| `/task-run` | 実装実行 | 各メンバーが担当Issueを実行 |

### チーム開発（並列実装）

複数レイヤーにまたがるタスク（フロント + バック + テスト等）は `/team-start` でエージェントチームを起動して並列開発する。

> **前提**: `claude --teammate-mode tmux` で起動していること。
> 通常モードで起動している場合は、一度 Claude Code を終了し、`claude --teammate-mode tmux` で再起動してから `/team-start` を実行する。

チーム運用の詳細は `doc/guide/team_protocol.md` を参照。

### テスト戦略（TDD）

- **単体テスト**: RED → GREEN → REFACTOR（テストライブラリはRDDで指定）
- **E2E/結合テスト**: Playwright（TDD的に書く）
  - 本番: 網羅
  - プロトタイプ: 主要フローのみ
- **デザイン確認**: Agent Browserで探索的チェック

### レビュー（チーム/シングル共通で徹底）

- `/basic-review` → 必要に応じて `/deep-review`
- レビュー結果はPRにコメント
- チーム時: GitHub（PR/Issue）必須
- 適宜ブランチを切る

## 4. ユーザー確認

- ユーザーが動作確認し、フィードバック・指摘を伝える

---

## 5. 指摘修正ループ

- フィードバック → §3と同じ開発フローで修正
- 適宜ブランチを切る
- 指摘がなくなるまで繰り返し

## 6. バグ修正（任意タイミングで発動）

開発中・ユーザー確認中・運用中を問わず、バグが発見されたら以下のフローで対応する。GitHub Issue（調査・議論）→ PR（修正）の流れ。

### フロー

```
/bug-new → /bug-investigate → /bug-propose → /bug-fix
```

| ステップ | スキル | やること | 成果物 |
|---------|--------|---------|--------|
| 1. 起票 | `/bug-new` | 問題の概要・再現手順・仮説を記録 | GitHub Issue（ラベル: `bug`） |
| 2. 調査 | `/bug-investigate` | Issue番号を指定して調査実施 | Issueコメントに調査結果を追記 |
| 3. 修正案 | `/bug-propose` | 修正案をIssueコメントに追記 | 合理的な修正策のリスト |
| 4. 修正実行 | `/bug-fix` | ブランチ作成 → 修正 → PR作成 | PR（`Fixes #<Issue番号>` でIssue紐づけ） |

### 判断基準

- **環境構築・設定系のバグ**: `/bug-propose` で修正案を整理してから `/bug-fix`
- **アプリのロジックバグ**: `/bug-propose` はスキップ可。タスク化（`/task-list`）して開発フローに乗せることも検討
- **本番緊急修正**: `hotfix/*` ブランチで対応（通常の `feature_fix/*` ではなく）

### ブランチ

- 通常: `feature_fix/<Issue番号>-<概要>`
- 緊急: `hotfix/<Issue番号>-<概要>`
